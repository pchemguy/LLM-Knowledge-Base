# ONBOARDING

## What this repo actually is

ΩmegaWiki is a **repo-local research workflow runtime**, not primarily a web app or a hosted knowledge system.

- durable state lives in `wiki/*.md` and `wiki/graph/*.jsonl`
- the data model lives in `runtime/schema/*.yaml`
- deterministic state operations live in `tools/research_wiki.py`
- consistency checks live in `tools/lint.py`
- workflow behavior mostly lives in Claude Code skill specs under `i18n/en/skills/*/SKILL.md`

The core mental model is: **LLM-orchestrated workflows operating on a typed local wiki database**.

## Core runtime model

### State ownership

- `raw/papers`, `raw/notes`, `raw/web`: user-owned inputs, treated as read-only
- `raw/discovered`, `raw/tmp`: tool-owned staging
- `wiki/<kind>/*.md`: canonical entity state
- `wiki/graph/*`: derived/tool-owned graph and context artifacts
- `wiki/log.md`: append-only operational log
- `wiki/index.md`: rebuildable catalog
- `.checkpoints/` or `wiki/.checkpoints/`: resumable workflow state

### Main deterministic engine

`tools/research_wiki.py` owns:

- wiki init
- slug generation
- log appends
- frontmatter reads/writes
- edge/citation writes + dedup
- context/gap rebuilds
- lifecycle transitions
- stats/maturity
- checkpoint persistence
- index rebuild

If behavior changes a wiki mutation primitive, this file is usually the place.

### Health enforcement

`runtime/policy/writers.yaml` is **not enforced at runtime**. Structural safety comes mainly from `tools/lint.py`, which checks required fields, enums, link integrity, xref symmetry, and graph/citation consistency.

## Execution model

### Canonical usage path

1. run `setup.sh --lang en` or `setup.ps1 -Lang en`
2. `claude login`
3. `claude`
4. `/setup`
5. add sources under `raw/`
6. `/init [topic]` or `/ingest <paper>`
7. `/ask`, `/check`, `/discover`, `/daily-arxiv`, etc.

### Important flows

- `/init`: prepare local sources -> plan discovery -> fetch -> scaffold wiki -> parallel `/ingest` via git worktrees -> merge -> dedup -> rebuild -> lint
- `/ingest`: resolve source -> enrich metadata -> write paper page -> dedup/create concepts/methods/people -> add graph/citations -> update topics/index/log -> rebuild derived context
- `/ask`: use `wiki/index.md` + `wiki/graph/context_brief.md` + selected pages to answer; may crystallize answers back into `wiki/outputs/`
- `/check`: run `tools/lint.py`; optional deterministic autofix
- `tools/serve.py`: local SPA/API only; it **does not run skills**, it synthesizes copyable slash commands through `/api/intent/*`

## Architectural constraints and invariants

- edit `i18n/en/CLAUDE.md`, not root `CLAUDE.md`; root is synced/generated by setup
- `wiki/graph/` must be modified through `tools/research_wiki.py`, not by hand
- `wiki/log.md` is append-only
- forward link -> reverse link is an explicit invariant from `runtime/schema/xref.yaml`
- `foundations` are terminal targets
- `/ingest` is intentionally narrower than `/check`; don’t re-implement the linter inside ingest logic
- `/init` parallelism relies on git worktrees plus `.gitattributes` `merge=union` for `wiki/log.md`, `wiki/index.md`, `wiki/graph/edges.jsonl`, `wiki/graph/citations.jsonl`

## Extension points

### Data model

Change:

- `runtime/schema/*.yaml`
- `runtime/templates/*.md.tmpl`

Only change Python when extending the schema language itself.

### Workflow behavior

Change:

- `i18n/en/skills/*/SKILL.md`
- `i18n/en/shared-references/*`

Then sync with `./setup.sh --lang en`.

### UI/API

Change:

- `tools/serve.py`
- `app/modules/api.js`
- `app/modules/*`

Be careful: `app/modules/schema.js` duplicates runtime schema constants, so schema changes may require frontend updates too.

## Sharp edges / known debt

- workflow permissions in `runtime/policy/writers.yaml` are advisory only
- multiple tools implement custom YAML/frontmatter parsing
- frontend duplicates schema metadata
- fresh checkout is only partially initialized: `wiki/graph/` is scaffold-only until `research_wiki.py init wiki/`
- repo claims strong testing in `CHANGELOG.md`, but no test suite is present in this checkout
- there are two `.env.example` files (`.env.example` and `config/.env.example`)
